# 心跳检测设计：三层防御体系与失活连接清理

在高性能网络服务器开发中，管理海量连接的生命周期、及时清理失活连接（Dead Connections）是保障服务器高可用性和防止资源耗尽的核心课题。本设计文档基于 **STAR 法则** 梳理 Tudou 网络库中的探活与空闲清理机制。

## 1. Situation（情景）

在真实复杂的网络环境中，TCP 连接可能会因为各种原因变成“死连接”或“僵尸连接”：
1. **硬故障**：客户端拔网线、断网、路由器宕机、对端操作系统崩溃（蓝屏/死机）。
2. **软故障（应用层异常）**：客户端机器和网络均正常，但客户端应用程序发生死锁、假死，或主线程被死循环卡住。
3. **恶意攻击**：恶意客户端（如 Slowloris 慢速攻击）或故障客户端与服务器建立连接后，不发送任何有效业务数据，或以极慢的速度（如每分钟 1 个字节）发送数据，达到“占着茅坑不拉屎”的目的。

如果不及时发现并清理这些连接，它们会白白耗尽服务器宝贵的文件描述符（FD）和内存资源，最终导致合法用户无法连入。

一个常见的开发误区是：“既然系统内核已经提供了 TCP Keep-alive 机制，应用层就不需要再写心跳检测了。” 但实际上，内核机制存在盲区，无法单独应对上述所有场景。

## 2. Task（任务）

设计一套**无死角、全覆盖且职责分明**的连接探活与清理架构，确保：
1. 能够准确识别网络层面的链路断开。
2. 能够准确识别业务层面的应用假死或恶意占坑。
3. 在检测到失活后，能够干净利落地将连接关联的应用层对象（`TcpConnection`、上下文等）和底层系统资源（Socket FD）彻底释放。

## 3. Action（行动）

为了完成上述任务，Tudou 采用了 **“内核探活 + 应用层被动空闲检测”** 互补的机制，并依赖 RAII 进行资源释放。这构成了一个严密的三层防御体系。

### 3.1 探活三层防御体系架构

| 防御层级 | 机制名称 | 位于哪里 | 负责检测什么 | 主要防御对象 |
| :--- | :--- | :--- | :--- | :--- |
| **第一层 (L4)** | **TCP Keep-alive (系统探活)** | 操作系统内核 | 链路连通性、对端操作系统是否存活 | 拔网线、断网、中间路由器宕机、对端 OS 崩溃 |
| **第二层 (L4/L7 边界)** | **被动空闲检测 (ConnectionHeartbeat)** | 应用层 TCP 网络库 (`tudou/tcp`) | 用户态连接是否活跃（是否持续有入站字节上浮到用户态） | 对端 App 死锁/假死、恶意占坑不发数据的僵尸客户端、资源耗尽攻击 |
| **第三层 (L7)** | **业务层主动心跳 (Ping/Pong)** | 业务层代码 (如 WebSocket / RPC / 自定义长连接协议) | 协议处理路径是否健康、计算网络延迟 (RTT) | 对端 App 逻辑卡顿、需要双向探活的复杂业务场景 |

### 3.2 第一层：内核 Keep-alive 的运作机制
Tudou 在 `TcpConnection` 创建时通过 `conn->set_keep_alive(true)` 委托给系统内核。
- **原理**：当 TCP 连接空闲达到 `TCP_KEEPIDLE`（如 60 秒）时，内核自动发送没有数据负载的 TCP 探测包（Probe packet）。如果在多次重试后对端均无 ACK 响应，内核认定连接已死。
- **局限性**：如果对端应用程序死锁，但其操作系统正常，对端内核依然会自动回复 ACK。此时 Keep-alive 认为连接极其健康，无法应对软故障。

### 3.3 第二层：用户态空闲检测 (ConnectionHeartbeat)
为了填补内核探活的盲区，我们在 TCP 层实现了纯被动的空闲超时检测。
- **原理**：`ConnectionHeartbeat` 不主动发包，而是依赖 `refresh()` 方法刷新最后活跃时间 `lastActiveTime_`。
- **核心防御点**：`refresh()` 仅在 `TcpConnection::on_read` 成功读到入站字节、并一路上浮到 `TcpServer::on_message` 时才会被触发。这意味着，如果对方只是操作系统活着（内核回复 ACK）但连接上长期没有任何用户态可见的入站数据，时间一到（如 60 秒），定时器 `check_timeout` 就会触发并强制踢除连接（`force_close()`）。
- **定位**：它关心的是“这条 TCP 连接上是否持续有入站字节上浮到用户态”，属于底层通用网络防护，而不是严格意义上的“业务健康检测”。

### 3.4 第三层：业务层主动心跳 (TODO：当前版本暂未实现)
虽然第一层和第二层已经能防范绝大多数网络与应用异常，但在要求极高的复杂业务场景下（如长连接 RPC、WebSocket 实时通信），往往还会引入第三层主动心跳：
- **机制**：由业务层代码（如协议解析层）主动、周期性地向对端发送探测包（Ping 包），并等待对端回复（Pong 包）。
- **核心目的**：
  1. **更深一层的协议/业务探活**：确保对端不仅“能接收底层字节”，而且至少“协议处理与心跳响应路径依然健康，没有完全卡死”。
  2. **网络质量监控**：通过一来一回的时间差，计算应用层真实的端到端网络延迟（RTT）。
- **当前状态**：Tudou 网络库目前专注于底层网络基础设施的健壮性，已经落地了 L4 和 L4/L7 边界的防御机制。L7 的主动心跳通常强耦合于具体的应用协议栈，因此留作后续在具体业务层（如扩展 WebSocket 模块）中实现；对于 HTTP/1.1 这类请求-响应协议，很多场景下更适合补充请求 deadline，而不是原生 Ping/Pong。

### 3.5 失活后的资源安全释放流转
当内核或应用层判定连接失活后，资源的释放流程严密自洽：
1. **Epoll 触发（内核与应用层的桥梁）**：当系统内核宣告连接死亡（例如 Keep-alive 探测多次超时未响应）时，内核协议栈会在底层强制释放该连接的传输控制块（TCB），并将该 Socket 描述符的状态标记为可读或异常。此时，注册在 Epoll 实例上的该 fd 会触发 `EPOLLIN`（因为对端异常断开或内核模拟 EOF 也会表现为可读状态）或 `EPOLLERR` / `EPOLLHUP` 事件。这会立刻唤醒阻塞在 `epoll_wait` 系统调用上的 `EpollPoller::poll()`。
2. **事件分发与读取**：`Channel::handle_events_with_guard()` 捕获到上述 Epoll 事件后，会按事件类型将其分发给 `TcpConnection::on_read`、`on_error` 或 `on_close`。如果走到读路径，程序尝试执行 `read` 或 `readv` 系统调用时，可能返回 `0`（内核模拟了 EOF），或者返回 `<0` 并伴随 `ETIMEDOUT`、`ECONNRESET` 等错误；在 Tudou 当前实现中，只要不是 `EAGAIN` / `EWOULDBLOCK`，都会进入关闭流程。
3. **触发关闭**：无论是读到 `0`、读写报错，还是直接收到错误/挂断事件，最终都会收敛到 `close_connection()`。
4. **对象移出与降引用**：请求路由到 `TcpServer::remove_connection`，将持有 `TcpConnection` 的 `shared_ptr` 从 `connections_` map 中 `erase` 擦除。
5. **完美析构**：`shared_ptr` 引用计数归 0，触发 `TcpConnection` 及其绑定的应用层上下文（如 `HttpContext`）析构。期间 RAII 封装的 `Socket` 对象随之析构，自动调用 `::close(fd)` 将文件描述符归还给操作系统。

## 4. Result（结果）

通过引入这套设计，Tudou 实现了极高健壮性的连接管理：
1. **互补不重叠**：内核 Keep-alive 零 CPU 消耗地拦截了物理层与系统层宕机场景；而应用层空闲检测（ConnectionHeartbeat）进一步拦截了长时间静默的僵尸连接，以及一部分应用层假死或占坑场景。
2. **内存与 FD 零泄漏**：得益于智能指针与 RAII，连接的断开操作（无论是被动感知还是主动强制断开），都会最终收敛到统一的 `erase` 与生命周期结束流程中，没有悬空指针或游离的 socket 描述符残留。
3. **架构清晰**：L4 的探活、L4/L7 的空闲检测与具体业务协议保持解耦；若后续确有需要，还可以在 L7 单独补上主动心跳或 deadline 机制。
