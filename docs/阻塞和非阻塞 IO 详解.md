# 阻塞 IO 与非阻塞 IO 详解

在网络编程、Linux 系统编程和高性能服务器开发里，阻塞 IO 与非阻塞 IO 是绕不开的基础题。它们本身不难，但面试里很容易被追问到细节，比如：

- 阻塞和非阻塞到底描述的是谁？
- 为什么阻塞 IO 明明简单，却还会被大量使用？
- 非阻塞 IO 为什么几乎是 Reactor 模式的标配？
- `epoll` 已经告诉你“可读了”，为什么还要把 fd 设成非阻塞？
- LT 和 ET 到底分别该怎么写代码？

本文按“它们是什么 -> 为什么使用 -> 如何使用 -> 面试怎么答”来讲，尽量把概念、行为和实践串起来。

---

## 1. 先把 3 个容易混淆的概念分开

很多人一上来就把“阻塞/非阻塞”“同步/异步”“`epoll`/多路复用”混在一起。面试时最好先分清。

### 1.1 阻塞 / 非阻塞

它描述的是：**当你发起一次 IO 系统调用时，如果当前条件还不满足，调用线程会不会睡眠等待。**

这里的“条件还不满足”举例来说就是：

- `read` 时接收缓冲区里还没有数据
- `write` 时发送缓冲区暂时写不进去
- `accept` 时还没有新连接到来
- `connect` 时连接还没有立即建立完成

如果线程会一直等到条件满足再返回，就是**阻塞 IO**。

如果线程不会睡眠，而是立刻返回一个“现在还不行”的结果，通常是 `EAGAIN`、`EWOULDBLOCK` 或 `EINPROGRESS`，就是**非阻塞 IO**。

### 1.2 同步 / 异步

它描述的是：**真正的数据读写完成这件事，是由谁在主导，以及完成后怎么通知你。**

- `read` / `write` / `accept` / `connect` 这类传统 socket IO，本质上还是**同步 IO**
- `epoll` 不是异步 IO，它只是告诉你“现在这个 fd 可以进行某种 IO 了”

这是一个高频陷阱题：

> `epoll + 非阻塞 socket` 不是“异步 IO”，而是“同步 IO + IO 多路复用 + 非阻塞 fd”。

### 1.3 IO 多路复用

`select`、`poll`、`epoll` 做的事情不是替你读写数据，而是：

**帮你同时盯着很多 fd，等哪个 fd 进入“可读/可写/异常”状态时，再通知你去做真正的 `read` / `write` / `accept` / `connect`。**

所以要记住：

- 阻塞/非阻塞：是 IO 调用本身的行为问题
- 多路复用：是“谁先就绪”的通知问题
- 同步/异步：是“数据拷贝完成由谁负责”的模型问题

## 2. 阻塞 IO 和非阻塞 IO 到底在描述什么？

最准确的说法不是“socket 是阻塞的”，而是：

> 某个文件描述符上的 IO 操作，在条件不满足时，是让调用线程睡眠，还是立即返回。

在 Linux 里，这通常通过 fd 的 `O_NONBLOCK` 状态标志来控制。常见做法是用 `fcntl` 给 socket、pipe、FIFO、terminal 等对象设置非阻塞。

要注意一个细节：

- `O_NONBLOCK` 对网络 socket、pipe、FIFO、终端这类对象非常重要
- 对**普通磁盘文件**和块设备，Linux 上 `O_NONBLOCK` 通常**没有实际效果**

所以在面试里，如果你谈阻塞 IO / 非阻塞 IO，默认场景通常是：

- TCP socket
- Unix domain socket
- pipe / FIFO
- 终端设备

而不是普通文件读写。

## 3. 一次 `read` / `write` 到底经历了什么？

理解阻塞和非阻塞，最好的方式是把一次 IO 调用拆开看。

### 3.1 `read` 在做什么

对一个 stream socket 来说，`read(fd, buf, n)` 大致有两件事：

1. 看内核接收缓冲区里有没有数据可读
2. 如果有，就把数据从内核空间拷贝到用户空间的 `buf`

如果第 1 步发现“现在没有数据”：

- 阻塞 fd：线程睡眠等待
- 非阻塞 fd：立即返回 `-1`，并设置 `errno = EAGAIN` 或 `EWOULDBLOCK`

如果有数据，那么无论阻塞还是非阻塞，`read` 都会直接返回已经读到的字节数。

还要记住两个重要返回值：

- `> 0`：成功读到数据
- `== 0`：对端关闭连接（对流式 socket 而言）

### 3.2 `write` 在做什么

对一个 stream socket 来说，`write(fd, buf, n)` 大致是：

1. 看内核发送缓冲区还有没有空间
2. 把用户数据拷贝到发送缓冲区，后续由协议栈慢慢发出去

如果发送缓冲区满了：

- 阻塞 fd：线程睡眠，等可写
- 非阻塞 fd：立即返回 `-1`，并设置 `errno = EAGAIN` 或 `EWOULDBLOCK`

无论阻塞还是非阻塞，都**不能假设一次 `write` 就把所有数据写完**。短写是正常情况，代码里应该维护“还有多少没写完”。

这也是面试里很容易被问到的一点：

> 阻塞 IO 不代表“每次 write 一定写满”，非阻塞 IO 也不代表“每次 write 都失败”。

## 4. 什么是阻塞 IO？为什么它仍然有价值？

### 4.1 定义

阻塞 IO 指的是：

> 当一次 IO 调用当前无法立即完成时，调用线程会被挂起，直到条件满足、发生错误、被信号打断，或者连接关闭。

例如：

- 阻塞 `read`：没数据就等
- 阻塞 `write`：写不进去就等
- 阻塞 `accept`：没有新连接就等
- 阻塞 `connect`：连接未建立完成就等

### 4.2 直观理解

可以把阻塞 IO 想成：

> 你给内核下了一个命令，然后你自己站在原地等结果，结果没出来之前，你什么都不干。

### 4.3 阻塞 IO 的优点

阻塞 IO 最大的优点不是“性能好”，而是：

- **控制流简单**：顺序写代码就行
- **心智负担小**：调用返回前，事情通常已经有结果了
- **更适合串行业务逻辑**：例如命令行工具、脚本、小型客户端、简单 RPC 调用、低并发后台任务

如果你的场景是：

- 并发连接数不高
- 每个线程只服务很少的连接
- 或者本来就有独立工作线程可以等待

那么阻塞 IO 完全是合理选择。

### 4.4 阻塞 IO 的问题

问题出在高并发场景。

假设你有 10 万个 TCP 连接，如果采用“一个连接一个线程”的阻塞模型，会遇到：

- 大量线程内存开销
- 频繁上下文切换
- 大多数线程都在睡眠等待 IO
- 任意一个慢连接都独占一个线程

所以阻塞 IO 的核心问题不是“不能工作”，而是：

> 当连接数很大时，它要求你为“等待”这件事付出过多线程资源。

## 5. 什么是非阻塞 IO？为什么高性能服务器偏爱它？

### 5.1 定义

非阻塞 IO 指的是：

> 当一次 IO 调用当前不能立即完成时，系统调用不会让线程睡眠，而是马上返回，让你稍后再试。

典型返回是：

- `read` / `write` / `accept` 的 `EAGAIN` 或 `EWOULDBLOCK`
- `connect` 的 `EINPROGRESS`

### 5.2 直观理解

可以把非阻塞 IO 想成：

> 你去问内核“现在能不能干这件事”，如果不能，内核立刻告诉你“现在不行”，你可以先去处理别的事情。

### 5.3 非阻塞 IO 的价值

非阻塞 IO 的真正价值不是“单次调用更快”，而是：

> 它保证一个线程不会因为某一个 fd 的等待而卡住。

这对事件循环线程非常关键。因为事件循环线程要服务很多连接，如果对其中一个连接执行一次阻塞 `read` 或阻塞 `write`，整个线程就会停住，其他连接也跟着饿死。

### 5.4 非阻塞 IO 单独使用的问题

如果你只是把 fd 设成非阻塞，然后这样写：

```cpp
while (read(fd, buf, sizeof(buf)) < 0 && errno == EAGAIN) {
    // 一直重试
}
```

那你只是把“线程睡眠等待”换成了“CPU 忙等轮询”。

这就是为什么：

> **单独使用非阻塞 IO 没有意义，真正高效的方式是“非阻塞 IO + IO 多路复用”。**

## 6. 为什么阻塞 IO 和非阻塞 IO 都会被使用？

这是面试里很值得主动回答的一点。不要把阻塞 IO 说成“落后方案”，也不要把非阻塞 IO 说成“绝对更先进”。

### 6.1 什么时候更适合阻塞 IO

阻塞 IO 通常更适合：

- 命令行程序、脚本、工具类程序
- 简单客户端
- 低并发服务
- 每个请求都有专属工作线程的系统
- 对开发效率和代码可读性要求更高、对并发规模要求不高的场景

一句话概括：

> **连接数不多、线程预算充足、逻辑以串行为主时，阻塞 IO 是简单且可靠的工程选择。**

### 6.2 什么时候更适合非阻塞 IO

非阻塞 IO 通常更适合：

- 高并发网络服务器
- Reactor 模式
- 一个线程管理很多连接的场景
- 需要事件循环、定时器、心跳、超时控制统一调度的系统

一句话概括：

> **当你不希望一个线程被单个连接拖住时，就应该考虑非阻塞 IO。**

## 7. 非阻塞 IO 如何使用？

### 7.1 如何把 fd 设为非阻塞

Linux 上常见做法是通过 `fcntl` 设置 `O_NONBLOCK`：

```cpp
#include <fcntl.h>

int flags = fcntl(fd, F_GETFL, 0);
fcntl(fd, F_SETFL, flags | O_NONBLOCK);
```

对于新连接，Linux 上还有一个很实用的细节：

- `accept()` 返回的新 socket **不会自动继承**监听 socket 的 `O_NONBLOCK`
- 更稳妥的做法是使用 `accept4(..., SOCK_NONBLOCK | SOCK_CLOEXEC)`，或者在 `accept()` 之后立即再设一次非阻塞

### 7.2 常见系统调用在非阻塞下的行为

| 调用 | 阻塞 fd 的行为 | 非阻塞 fd 的行为 |
|---|---|---|
| `read` | 没数据时等待 | 没数据时返回 `EAGAIN` / `EWOULDBLOCK` |
| `write` | 写缓冲满时等待 | 写不进去时返回 `EAGAIN` / `EWOULDBLOCK` |
| `accept` | 没新连接时等待 | 没连接时返回 `EAGAIN` / `EWOULDBLOCK` |
| `connect` | 连接建立完成前等待 | 立即返回 `EINPROGRESS`，后续等待可写事件 |

### 7.3 非阻塞 `connect` 正确姿势

很多人只记住了“返回 `EINPROGRESS`”，但没记住后半句。

正确流程通常是：

1. 调用 `connect`
2. 如果返回 `0`，说明立刻连上了
3. 如果返回 `-1` 且 `errno == EINPROGRESS`，说明连接正在建立中
4. 用 `select` / `poll` / `epoll` 等待该 socket **变为可写**
5. 可写后不能直接认定成功，要用 `getsockopt(fd, SOL_SOCKET, SO_ERROR, ...)` 检查是否真的连上

这是非阻塞连接题里最容易丢分的点。

### 7.4 非阻塞代码要接受“暂时做不了”是正常流程

在阻塞模型里，“现在做不了”往往被内核替你等待掉了。

在非阻塞模型里，“现在做不了”会显式地回到你的代码里，所以你必须把这些返回值当成正常控制流，而不是异常路径：

- `EAGAIN`
- `EWOULDBLOCK`
- `EINPROGRESS`

## 8. 为什么非阻塞 IO 几乎总是和 `epoll` 一起出现？

因为单独的非阻塞 IO 解决不了“什么时候该再试一次”的问题。

`epoll` 解决的是这个问题：

> 帮你高效地从成千上万个 fd 里找出“现在值得处理的那几个”。

### 8.1 `epoll` 告诉你的到底是什么

一个非常精确、很值得面试时说出来的表述是：

> `select` / `poll` / `epoll` 报告的是“ready”，也就是**如果此刻对这个 fd 执行对应 IO，那么在阻塞模式下它本不该阻塞**。

这说明两点：

1. `epoll` 本身不做数据传输，它只做“就绪通知”
2. “就绪”是一个时刻性的状态，不代表你后续逻辑里就一定永远不会阻塞

尤其在多线程、共享 fd 或复杂状态切换场景下，从 `epoll_wait` 返回到你真正执行 `read` / `accept` 之间，状态可能已经变了。所以高性能服务器通常仍然坚持把 fd 设成非阻塞，确保“最多拿到一个 `EAGAIN`，而不是把事件循环卡死”。

### 8.2 Reactor 模式为什么离不开它

Reactor 模式的典型工作流是：

1. 用 `epoll_wait` 阻塞等待事件
2. 拿到一批就绪 fd
3. 对这些 fd 做非阻塞 `read` / `write` / `accept`
4. 没有事件时继续阻塞在 `epoll_wait`

这里真正阻塞的是：

- 阻塞在 `epoll_wait`

而不是：

- 阻塞在某个业务连接的 `read` / `write`

这就是为什么 Reactor 能用少量线程服务大量连接。

## 9. `epoll` 的 LT 和 ET 到底怎么理解？

`epoll` 有两种核心触发语义：

- LT：Level Triggered，水平触发
- ET：Edge Triggered，边沿触发

### 9.1 LT：水平触发

LT 是默认模式。它的语义是：

> 只要 fd 仍然处于“可读/可写”状态，`epoll_wait` 就可能继续把它返回给你。

例如读事件：

- 某个 socket 收到了 2 KB 数据
- 你只读走了 1 KB
- 还有 1 KB 留在内核缓冲区
- 下次 `epoll_wait`，这个 fd 还会继续被报告为可读

所以 LT 的特点是：

- 更宽容
- 更容易写对
- 漏读一次通常问题不大，因为后面还会继续通知

#### LT 一定要配非阻塞吗？

**不是强制。**

从语义上看，LT 和 `poll` 类似，可以配合阻塞 fd 使用。

但是在通用网络库和高性能服务器里，通常仍然会：

> **LT + 非阻塞 fd**

原因不是“LT 用阻塞一定错”，而是：

- 统一整套网络编程范式
- 避免某个边缘条件把事件循环卡住
- 避免 `accept` / `read` / `write` 在状态变化后意外阻塞
- 更容易支持多线程、回调链和复杂状态机

一句话总结：

> LT 可以配阻塞，也可以配非阻塞；工程上更常见的是 LT + 非阻塞。

### 9.2 ET：边沿触发

ET 的语义是：

> 只有当 fd 从“不就绪”变成“就绪”时，你才会收到通知。

还是读事件的例子：

- 缓冲区原来没数据
- 现在来了 2 KB 数据，状态从“不可读”变成“可读”
- 你收到一次事件
- 如果你只读了 1 KB，缓冲区里还剩 1 KB
- 但只要没有新的“状态变化”，你可能**不会再次收到通知**

这就是 ET 最关键的特点：

- 通知次数更少
- 对程序正确性要求更高
- 如果你没把这次该处理的数据处理干净，很容易“饿死”这个 fd

### 9.3 为什么 ET 必须配非阻塞 IO

这是面试必答点。

在 ET 下，你收到可读事件后，通常要循环读取，直到读到：

- `EAGAIN`
- `EWOULDBLOCK`
- 或者返回 `0`（对端关闭）

原因是：你必须尽量把本轮可处理的数据“吃干净”，否则后面可能收不到再次提醒。

如果这时 fd 是阻塞的，就会出现致命问题：

- 你在循环里不断 `read`
- 读到最后，缓冲区已经空了
- 下一次 `read` 会等待新数据到来
- 整个事件循环线程被卡死

所以在 ET 模式下：

> **非阻塞不是建议，而是基本要求。**

### 9.4 ET 下的经典读取逻辑

```cpp
while (true) {
    ssize_t n = read(fd, buffer, sizeof(buffer));

    if (n > 0) {
        // 处理读到的数据
        continue;
    }

    if (n == 0) {
        // 对端关闭连接
        break;
    }

    if (errno == EINTR) {
        continue;
    }

    if (errno == EAGAIN || errno == EWOULDBLOCK) {
        // 本轮已经读干净了
        break;
    }

    // 其他错误
    break;
}
```

### 9.5 ET 下别忘了 `accept` 也要循环

对监听 socket 也是一样。

如果监听 fd 用 ET，某一时刻可能有多个连接同时进入 accept 队列。你不能只 `accept` 一个就停下，否则队列里剩下的连接可能暂时收不到下一次提醒。

正确做法通常是：

- 循环 `accept`
- 直到返回 `EAGAIN` / `EWOULDBLOCK`

## 10. 阻塞 IO 与非阻塞 IO 的本质区别

如果面试官让你一句话概括，可以这样说：

> 阻塞和非阻塞的本质区别，不在于“数据最终怎么来”，而在于“当这次 IO 现在还做不了时，调用线程是睡眠等待，还是立即返回”。

把它展开一点，就是：

- 阻塞 IO：把等待藏在内核里
- 非阻塞 IO：把“现在做不了”显式返回给用户态

所以两者的权衡也非常清楚：

- 阻塞 IO 换来简单控制流
- 非阻塞 IO 换来更强的并发调度能力

## 11. 常见误区与易错点

### 11.1 “非阻塞 IO 就是异步 IO”

错。

非阻塞只是“当前做不了时立即返回”，数据收发仍然由你的代码自己在合适时机调用 `read` / `write` 完成，所以它仍属于同步 IO 范畴。

### 11.2 “`epoll` 已经通知可读了，所以 fd 阻塞也没关系”

不严谨。

从语义上，LT 可以配阻塞 fd；但工程上这样做有风险。因为从事件返回到你实际操作之间，状态可能变化，或者你的代码为了“多读一点”进入额外读取逻辑，从而卡住线程。高性能框架通常统一设非阻塞，以保证事件循环不会因为某次 IO 被拖住。

### 11.3 “只有 `read` 会阻塞，`write` 不会”

错。

对 socket 来说，发送缓冲区满时，阻塞 `write` 一样会卡住线程。

### 11.4 “一次 `read` / `write` 一定读满 / 写满”

错。

短读、短写都是正常现象。特别是在流式协议中，应用层必须自己管理消息边界和剩余数据。

### 11.5 “`EAGAIN` 是异常，说明程序出错了”

错。

在非阻塞模型里，`EAGAIN` / `EWOULDBLOCK` 往往只是说明：

> 现在先别做这件事，等下一次可读/可写事件。

它们属于正常控制流。

### 11.6 “非阻塞 `connect` 可写就一定连成功了”

错。

socket 可写只说明“连接过程结束了”，不保证成功。必须再检查 `SO_ERROR`。

### 11.7 “普通文件也很适合拿来讨论非阻塞 IO”

不合适。

在 Linux 上，普通文件上的 `O_NONBLOCK` 通常没有实际意义。阻塞/非阻塞最核心的讨论对象还是 socket、pipe、FIFO、终端等可等待对象。

## 12. 面试时怎么答最稳

### 12.1 30 秒版本

阻塞和非阻塞描述的是 IO 系统调用在当前无法立即完成时，调用线程会不会等待。阻塞 IO 会让线程睡眠，直到数据就绪或缓冲区可写；非阻塞 IO 会立即返回 `EAGAIN`、`EWOULDBLOCK` 或 `EINPROGRESS`。阻塞 IO 编程简单，适合低并发或专属线程场景；非阻塞 IO 适合高并发服务器，通常要和 `epoll` 结合，避免线程被单个连接卡住。

### 12.2 60 秒版本

我会先区分三个概念：阻塞/非阻塞说的是调用线程会不会等待，同步/异步说的是数据完成方式，`epoll` 只是多路复用的就绪通知。阻塞 IO 的优势是代码简单，但高并发下会造成大量线程等待；非阻塞 IO 的优势是线程不会被单个 fd 卡住，但必须和 `epoll` 这类机制配合，否则会忙等。`epoll` 的 LT 模式更宽容，工程上常见的是 LT 加非阻塞；ET 模式要求更严格，必须配非阻塞，并且读写或 `accept` 时循环到 `EAGAIN` 为止，否则容易漏事件或卡死事件循环。

### 12.3 如果面试官继续追问“为什么高性能服务器都用非阻塞？”

你可以补一句：

> 因为高性能服务器的关键不是让单个连接更快，而是让一个线程能安全地管理很多连接，不被任何一个慢连接阻塞住。

## 13. 最后给一个总表

| 维度 | 阻塞 IO | 非阻塞 IO |
|---|---|---|
| 当前做不了时 | 线程睡眠等待 | 立即返回 |
| 编程难度 | 低 | 高 |
| 线程利用率 | 低 | 高 |
| 是否适合高并发事件循环 | 不适合 | 适合 |
| 是否容易忙等 | 不会 | 单独使用时会 |
| 常见搭配 | 线程池、串行逻辑 | `epoll` / Reactor |
| 面试关键词 | 简单、直观、线程换等待 | 就绪通知、事件循环、`EAGAIN` |

---

## 14. 一句话收尾

如果你只记一句话，请记住：

> **阻塞 IO 把等待留在内核里，非阻塞 IO 把“暂时做不了”显式交还给用户态；高性能服务器用非阻塞 IO，不是因为它神奇更快，而是因为它让一个线程不会被某一个连接拖死。**
