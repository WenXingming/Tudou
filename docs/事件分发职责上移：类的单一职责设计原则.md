# 事件分发职责上移：类的单一职责设计原则（STAR 法则）

## Situation（情景）
Tudou 是一个基于 Reactor 模式的多线程 C++ 网络框架。在该框架中，`EventLoop` 是 Reactor 的核心调度器，而 `EpollPoller` 是对 epoll 的底层封装。在重构之前的实现中，`EpollPoller::poll()` 方法不仅负责等待和获取就绪事件，还将这些事件翻译为 `Channel` 列表，并**直接负责将事件分发到具体的 Channel 中**（调用 `Channel::handle_events()`），同时还要调节缓冲区容量。这种设计导致分发行为过度内聚在底层机制 `EpollPoller` 中，使其越过了作为底层封装的职责边界，违反了类的单一职责设计原则（SRP）。

## Task（任务）
为了使代码结构更加清晰，符合单一职责设计原则，并且更好地体现 Reactor 模式的架构设计，我们需要将事件分发的职责从底层的 `EpollPoller` 上移到核心调度器 `EventLoop` 中。
具体目标为：
1. 使 `EpollPoller` 专注于 epoll 机制的纯粹封装（包括创建、注册、等待、翻译就绪事件以及容量调节）。
2. 让 `EventLoop` 接管 Reactor 的核心策略行为，即：获取就绪列表 $\rightarrow$ 遍历分发事件 $\rightarrow$ 执行待处理任务。

## Action（行动）
为了实现这一职责的分离，我们进行了以下技术方案的设计与重构：

### 1. 接口重构与数据流向调整
我们将 `EpollPoller::poll()` 的签名从 `void` 更改为返回 `std::vector<Channel*>`。该函数现在只负责收集并返回活跃的 `Channel` 列表，不再直接调用处理逻辑。
随后，在 `EventLoop::loop()` 主循环中，接管返回的活跃列表，并负责遍历调用每个 Channel 的处理函数：

```cpp
// EventLoop::loop() — 分发逻辑回归 EventLoop
auto activeChannels = poller_->poll(timeoutMs);
for (Channel* channel : activeChannels) {
    channel->handle_events();
}
do_pending_functors();
```

### 2. 梳理并明确类的职责对比
重构前后，各层职责发生明确转变，严格践行了单一职责原则：
- **重构前：** `EpollPoller` 承担了 wait + translate + **dispatch** + resize 的职责；`EventLoop` 仅仅是调用 `poller_->poll()`。
- **重构后：** `EpollPoller` 仅负责 wait + translate + resize；而 `EventLoop` 负责通过 poll 获取列表 + **dispatch** + do_pending_functors。

### 3. 设计原则的落地（单一职责与相邻通信）
- **单一职责（SRP）：** `EpollPoller` 作为 I/O 复用的**机制**抽象，不应感知上层 `Channel::handle_events()` 的接口细节。事件分发是 Reactor 的**策略**行为，理应归属于 `EventLoop` 的调度范畴。
- **相邻类通信原则：** `EventLoop` 直接持有 `EpollPoller` 和 `Channel` 两者的引用，由它作为桥梁完成“从底层机制获取数据 $\rightarrow$ 通知上层组件”的交互，维持了清晰的分层逻辑。

## Result（结果）
此次重构带来了以下显著效果：
1. **职责清晰，接口语义明确：** `EpollPoller` 的公开接口 `poll()` 移除了副作用，其返回值直白地表达了“收集就绪 Channel”的能力。它不再依赖 `Channel::handle_events()`，大大降低了上下层之间的耦合度。
2. **提升代码可读性与架构一致性：** `EventLoop::loop()` 真正成为了 Reactor 流程的唯一可读入口（poll $\rightarrow$ dispatch $\rightarrow$ pending functors），开发者无需深入 `EpollPoller::poll()` 内部即可看清整个事件循环的脉络。
3. **零性能损耗：** 得益于 C++ 的返回值优化（NRVO），返回 `std::vector<Channel*>` 并没有引入额外的拷贝开销，在提升架构合理性的同时保障了系统性能。
4. **易于测试与扩展：** 下层提供能力（收集 Channel），上层决定策略（如何分发）。如果未来需要实现优先级分发或批量分发，只需在核心调度器 `EventLoop` 中修改即可，无需改动底层 epoll 机制的封装。
