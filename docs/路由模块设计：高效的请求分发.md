# HTTP 路由模块设计：为什么我们需要一个专门的 Router？（STAR 法则）

本文档采用 STAR (Situation, Task, Action, Result) 法则，阐述在开发 Tudou 高性能 C++ HTTP 服务器时，设计并实现独立 `Router`（路由器）模块的思考过程。这有助于在面试中展示良好的架构解耦能力和对 HTTP 协议规范的深刻理解。

## 1. Situation (情境)
在服务器开发的前期，我已经完成了底层的非阻塞网络层（基于 Reactor 模式），并引入了 `llhttp` 将字节流成功解析为结构化的 `HttpRequest` 对象。
随着上层应用（如 `FileLinkServer` 文件服务、`StarMindServer` AI 对话服务）功能的不断丰富，服务器需要处理五花八门的请求路径和方法（例如：处理登录的 `POST /login`，获取页面的 `GET /`，下载文件的 `GET /file/{id}` 等）。
如果没有任何路由管理机制，这些请求的分发逻辑将不得不在网络层或核心的 `on_http_request` 回调中堆砌成百上千行的 `if-else` 或 `switch` 语句。这会导致代码极度臃肿、耦合严重、且极其难以维护。

## 2. Task (任务)
我需要设计并实现一个**独立、高效且语义完备的 HTTP 路由模块 (Router)**。我对该模块有以下核心诉求：
- **彻底解耦**：将“请求长什么样（协议解析）”和“请求该谁管（业务分发）”彻底分开。
- **灵活的匹配策略**：既要支持精确的 API 路由（如 `/api/login`），又要支持前缀路由，以便兜底处理静态资源或动态路径（如 `/static/` 或 `/file/`）。
- **严格遵循 HTTP 语义**：除了常规的匹配成功和 `404 Not Found` 之外，当请求的路径存在但 HTTP 方法错误时（例如对一个只允许 GET 的接口发起了 POST 请求），必须能规范地返回 `405 Method Not Allowed`。
- **分发性能优异**：路由查找发生在每个 HTTP 请求的关键路径上，必须尽可能的快。

## 3. Action (行动)
参考业界主流 Web 框架的设计思路，我从零实现了一个 `Router` 类，并采取了**多级分发、精准降级**的设计策略：

1. **O(1) 的精确匹配优先**：
   - 设计了以 `(Method, Path)` 为联合主键（`RouteKey`），结合自定义哈希算法的 `unordered_map`。
   - 当请求到来时，优先进行哈希查找。这保证了绝大部分明确的 API 请求能以 **O(1)** 的时间复杂度极快地命中对应的业务回调（Handler）。

2. **规范的 405 语义判定**：
   - 巧妙地单独维护了一个 `Path -> AllowedMethods` 的映射索引。
   - 如果精确匹配未命中，路由器会首先查询该路径是否被注册过其他方法。如果是，则立即拦截请求并自动生成标准的 `405` 响应，同时附带正确的 `Allow` 响应头（如 `Allow: GET, POST`），而不是粗暴地返回 404。

3. **有序的前缀兜底机制**：
   - 如果精确路由和 405 判定都未命中，最后会遍历前缀路由表。
   - 前缀路由表使用 `std::vector` 保存，以严格维护用户注册时的顺序（体现优先级）。这非常适合用来挂载前端静态文件目录或作为动态 ID 的解析入口。

4. **现代化的 API 设计**：
   - 对外暴露了高度语义化的注册接口（如 `add_get_route`、`add_post_route`），利用 C++11 的 `std::function` 和 Lambda 表达式，让业务侧可以非常内聚、直观地绑定回调函数。

## 4. Result (结果)
引入 `Router` 模块后，整个项目架构获得了质的提升：
- **极致的关注点分离（SoC）**：`HttpServer` 完全不知道业务长什么样，只负责把 `HttpRequest` 抛给 `Router`。上层应用（如 StarMind）只需在初始化时集中注册路由表即可。
- **完美的开闭原则（OCP）**：现在要在项目中新增一个 API 端点，只需一行 `router.add_post_route(...)`，完全不需要修改现有的核心调度逻辑，极大降低了引入 Bug 的风险。
- **专业规范的 HTTP 体验**：完善的 404 和 405 状态码处理，让服务器在应对标准 HTTP 客户端（如浏览器、Postman、前端 axios）时显得更加健壮和符合 RESTful 规范。

---

### Tips：
1. **强调 405 状态码的设计**：很多初中级开发者写路由只知道 200 和 404。如果你能在面试中主动提到“我特意为了 405 Method Not Allowed 维护了额外的路径索引并生成 Allow 头”，这会向面试官强烈暗示你**对 HTTP 协议/RESTful 规范有着远超平均水平的深入理解**。
2. **展现数据结构在工程中的权衡**：
   - 当面试官问到性能时，可以说：“对于精确路由，我采用了基于哈希表的 O(1) 查找追求极致性能；但对于前缀路由，因为需要依赖注册顺序决定优先级（比如先注册的具体前缀优于后注册的泛前缀），所以我权衡后选择了 `std::vector` 进行 O(N) 线性扫描作为兜底。这种**冷热分级**的设计符合实际的 Web 流量特征。”
