# 高水位回调：流量控制和自我保护机制设计

在非阻塞网络编程中（例如基于 Reactor 模式的网络库），`highWaterMarkCallback_`（高水位回调）是一项非常关键的流量控制和自我保护设计。以下将采用 STAR 法则（Situation情境、Task任务、Action行动、Result结果）来详细剖析该机制的设计。

## 1. Situation (情境)
在非阻塞 I/O 中，当服务器频繁向对端（客户端）发送数据时，如果操作系统的 TCP 发送缓冲区已满，后续的数据不会导致线程阻塞，而是由应用层接管，追加并积压在 `TcpConnection` 的应用层写缓冲区（`writeBuffer_`）中，等待可写事件触发时再逐步发送。
**痛点**：如果遇到接收速度极其缓慢的“慢客户端”（Slow Client），或者网络极度拥堵，而服务器业务逻辑仍在疯狂地向该连接 `send()` 数据（例如连续发送大文件），数据会大量堆积在应用层缓冲区中，导致服务器内存占用急剧飙升，最终可能引发 OOM (Out Of Memory) 导致服务直接崩溃。

## 2. Task (任务)
为了避免上述的 OOM 风险，网络库必须提供一种**告警和保护机制**。当应用层积压的未发送数据达到危险程度时，必须能够及时通知业务层停止生产数据，或者直接干预并断开恶意/异常积压的连接，以保护服务器内存资源。

## 3. Action (行动)
引入了“高水位回调”（`HighWaterMarkCallback`）机制，其核心执行逻辑如下：

1. **设置高水位阈值 (`highWaterMark_`)**：
   在 `TcpConnection` 中配置一个高水位线，默认为例如 64 MB (`64 * 1024 * 1024` 字节)。该阈值代表了系统允许单个连接在应用层积压数据的最大容忍度。
2. **边缘触发的越界检测**：
   在 `send()` 函数中，每次将数据追加写入 `writeBuffer_` 时，都会进行一次状态检查：
   ```cpp
   const size_t oldLen = writeBuffer_->readable_bytes();
   writeBuffer_->write_to_buffer(msg);
   const size_t newLen = writeBuffer_->readable_bytes();
   
   // 触发条件：以前的长度 < 高水位，且写入后的新长度 >= 高水位
   if (highWaterMarkCallback_ && oldLen < highWaterMark_ && newLen >= highWaterMark_) {
       handle_high_water_mark_callback();
   }
   ```
   **关键设计在于边缘触发**：只有在跨越该阈值的那一瞬间，才会触发回调，避免了数据持续在高水位之上时回调被无限次频繁调用。
3. **结合低水位回调 (写完成回调)**：
   通常在业务端，收到 `highWaterMarkCallback_` 后，会选择停止接收新请求或暂停向该连接发送数据；同时配合 `WriteCompleteCallback`（写完成回调/低水位回调），当积压的数据被逐渐清空后，再次恢复数据生产流程，实现一个动态的流量控制闭环。

## 4. Result (结果)
通过高水位回调设计，系统获得了**感知和干预极端积压**的能力。
1. **防止 OOM**：即便遭遇极其恶劣的网络环境或有意发起的慢客户端攻击，服务器内存也不会被单个连接无限消耗。
2. **平滑限流**：允许应用层建立反压 (Backpressure) 机制，实现生产者与消费者速率的动态平衡，大幅提升了服务器和网络库在复杂运行环境下的健壮性。
